Plan 01 + 02 outputs — same as Plan 03 interfaces section. Key additions for trajectory generation:

Trajectory message patterns (mlx-lm `tools` format):

**Single-turn (4 messages):**
```json
[
  { "role": "system", "content": "..." },
  { "role": "user", "content": "Show me the schema for the profiles table" },
  { "role": "assistant", "content": "", "tool_calls": [{ "id": "call_0", "type": "function", "function": { "name": "supabase_select_query_builder", "arguments": "{\"table\":\"profiles\",\"columns\":\"*\",\"limit\":1}" } }] },
  { "role": "tool", "content": "{\"result\":...}", "tool_call_id": "call_0", "name": "supabase_select_query_builder" },
  { "role": "assistant", "content": "The profiles table has columns: id (uuid), name (text), ..." }
]
```

**Multi-turn (6+ messages):** User asks -> assistant calls tool -> tool responds -> assistant answers -> user follows up -> assistant calls different tool -> ... Each turn validated independently.

**Parallel/dependent (tool_calls array has 2+ entries):**
```json
{ "role": "assistant", "tool_calls": [
  { "id": "call_0", "type": "function", "function": { "name": "tool_A", "arguments": "..." } },
  { "id": "call_1", "type": "function", "function": { "name": "tool_B", "arguments": "..." } }
]}
```
Followed by two `role: 'tool'` messages, one per call_id. Dependent: second call's arguments reference first call's result.

**Refusal (3 messages, NO tool_calls):**
```json
[
  { "role": "system", "content": "..." },
  { "role": "user", "content": "What's the current Supabase stock price?" },
  { "role": "assistant", "content": "I can't look up live stock prices — that would require network access which isn't available offline. However, I can help you with Supabase database queries, RLS policies, or edge functions." }
]
```

Tool selection for trajectories:
- Read `data/adapter-tools.json` at init time.
- For single-turn: randomly select 1 tool per example. Bias toward under-represented tools to help DAT-07 stratification.
- For multi-turn: randomly select 2-3 distinct tools per example.
- For parallel/dep: select 2 tools — parallel if independent, dependent if one logically feeds the other (e.g. query_builder result feeds into rls_policy_template).
- For refusal: select a scenario where the user asks for something that requires network or is out of scope.

Zod schemas for each trajectory type:
- Single-turn: `{ userQuery, toolCall: { name, arguments }, toolResult, assistantAnswer }`
- Multi-turn: `{ turns: Array<{ role: 'user'|'assistant'|'tool', content, toolCall?, toolResult? }> }`
- Parallel/dep: `{ userQuery, toolCalls: Array<{ name, arguments }>, toolResults: Array<unknown>, assistantAnswer, dependency: 'parallel'|'dependent' }`
- Refusal: `{ userQuery, refusalResponse }`
